## 3. Findings detail

### 1a — Full ICU / IANA timeZone in workerd — CONFIRMED

workerd is built on V8 with full-ICU compiled in. `Intl.DateTimeFormat` accepts any canonical IANA zone (`America/Sao_Paulo`, `Europe/Madrid`, `America/New_York`, etc.) in both `wrangler dev` and production. No compatibility flag is required for `Intl`; it is part of the standard global scope (CF "JavaScript and web standards" doc lists Intl as supported).

- **Caveat worth knowing (not blocking):** the *bundled ICU tzdata* is a fixed snapshot tied to the workerd/V8 build, so very recent IANA rule changes (e.g. the IANA **2026a** America/Vancouver permanent-DST change) can lag until CF ships a runtime update. For Inceptio this is immaterial: the cap is a soft cost-shaping counter and `reset_at_unix` is display-only — a zone whose rules changed last quarter would at most mis-display "resets tomorrow" by an hour for affected users, never mis-bucket the date in a way that breaks enforcement.
- **`wrangler dev` gotcha (relevant to the test gate):** local workerd runs with **`TZ=UTC`** so the *ambient* `Date`/`Intl` default zone is UTC regardless of the dev machine. This does **not** affect explicit-`timeZone` calls (which is all the spec uses) — but it means any test that accidentally relies on the host's local zone will behave differently locally vs. CI. Always pass an explicit `timeZone`; never rely on the default. The spec already computes the date from an explicit tz, so it's safe.

### 1b — `'en-CA'` → `YYYY-MM-DD` — CONFIRMED

`en-CA` with `{ year:'numeric', month:'2-digit', day:'2-digit' }` produces ISO-ordered `2026-06-06`. This is standard ECMA-402 behavior and stable across ICU versions. Acceptable. (Even-more-defensive alternative if you ever distrust locale drift: build the string from `formatToParts` and reassemble `${y}-${m}-${d}` — but `en-CA` is the idiomatic, widely-used shortcut and is fine here.)

### 1c — UTC offset via `timeZoneName:'shortOffset'` — CONFIRMED

`Intl.DateTimeFormat(locale, { timeZone, timeZoneName:'shortOffset' })` and `formatToParts` are fully supported in V8/workerd. `shortOffset` yields `GMT-3`/`GMT+5:30`; `longOffset` yields `GMT-03:00` (zero-padded, parse-friendly). For programmatic parsing prefer **`longOffset`** — its `±HH:MM` form needs no normalization, whereas `shortOffset` can emit `GMT-3` (no padding) and bare `GMT` for UTC.

### 1d — Next-local-midnight → unix, Intl-only, DST-correct — CAVEAT + canonical snippet

It is achievable with `Intl` alone (no date lib needed). **The pitfall** the spec must avoid: computing midnight by taking *today's* offset and subtracting it is wrong across a DST transition, because the offset *at midnight* may differ from the offset *now*. The robust technique computes the zone offset **at the candidate instant** and iterates once to settle on the boundary. Canonical, dependency-free snippet:

```ts
// Offset (minutes, east-positive) of `tz` at a given UTC instant.
// Uses the "format the instant as if in tz, interpret those wall-clock
// parts as UTC, diff against the real instant" trick — robust across DST.
function tzOffsetMinutes(utcMs: number, tz: string): number {
  const dtf = new Intl.DateTimeFormat('en-US', {
    timeZone: tz,
    year: 'numeric', month: '2-digit', day: '2-digit',
    hour: '2-digit', minute: '2-digit', second: '2-digit',
    hour12: false,
  });
  const p = Object.fromEntries(
    dtf.formatToParts(new Date(utcMs)).map((x) => [x.type, x.value]),
  );
  // 'hour' can be '24' at midnight in some ICU builds — normalize.
  const hour = p.hour === '24' ? '00' : p.hour;
  const asUTC = Date.UTC(
    +p.year, +p.month - 1, +p.day, +hour, +p.minute, +p.second,
  );
  return Math.round((asUTC - utcMs) / 60000);
}

// Local calendar date (YYYY-MM-DD) of `nowMs` in `tz`.
function localDate(nowMs: number, tz: string): string {
  return new Intl.DateTimeFormat('en-CA', {
    timeZone: tz, year: 'numeric', month: '2-digit', day: '2-digit',
  }).format(new Date(nowMs)); // en-CA => YYYY-MM-DD
}

// Unix seconds of the NEXT local midnight in `tz` after nowMs.
// DST-safe: compute the wall-clock midnight, convert using the offset
// AT that candidate instant, then re-check once (offset may shift across
// the boundary on a spring-forward/fall-back day).
function nextLocalMidnightUnix(nowMs: number, tz: string): number {
  const [y, m, d] = localDate(nowMs, tz).split('-').map(Number);
  // candidate: tomorrow 00:00 wall-clock, naive UTC
  const naiveTomorrowUTC = Date.UTC(y, m - 1, d + 1, 0, 0, 0);
  // first estimate using offset at the naive instant
  let offset = tzOffsetMinutes(naiveTomorrowUTC, tz);
  let utc = naiveTomorrowUTC - offset * 60000;
  // settle once: re-read offset at the estimated instant
  const offset2 = tzOffsetMinutes(utc, tz);
  if (offset2 !== offset) utc = naiveTomorrowUTC - offset2 * 60000;
  return Math.floor(utc / 1000);
}
```

Notes for the implementer (load-bearing):
- The "format-as-tz, reinterpret-as-UTC" offset trick is the standard dependency-free method and is what `date-fns-tz`/`Temporal` polyfills do internally. Confirmed to work in workerd (V8 `formatToParts`).
- The **one-iteration settle** handles the fall-back/spring-forward day where the offset at naive-tomorrow differs from the offset at the corrected instant. For a *daily-midnight* boundary this single re-check is sufficient; nonexistent/ambiguous wall-times only occur at the ~02:00 transition, not at 00:00, so midnight itself is never the skipped/duplicated hour in any current IANA zone — but the settle still corrects the *offset used*, which is what matters for the unix value.
- The spec already scopes `reset_at_unix` as **display/`Retry-After` only** and mandates a DST-boundary test. That scoping is correct and de-risks 1d. Keep the explicit DST test (e.g. `America/Sao_Paulo` historically, or a current DST zone like `Europe/Madrid`/`America/New_York` around the March/Nov transitions).

### 2a — KV `expirationTtl` minimum 60s vs. "seconds-to-next-local-midnight" — CAVEAT (real bug risk)

Confirmed from CF KV docs: **`expirationTtl` minimum is 60 seconds**; values targeting <60s in the future are rejected (the `put` rejects/throws). Also confirmed: a ~24h TTL is perfectly fine (max is far higher; 86,400s is unremarkable).

**The spec's TTL = "seconds until next local midnight (+ small buffer)" can drop below 60s** for any request arriving in the final minute before local midnight. At, say, 23:59:30 local, raw seconds-to-midnight = 30 → below the 60s floor → **`put` throws**. With telemetry/metric writes wrapped in `waitUntil`-with-swallowed-errors this might pass silently, but the **counter `put` is on the request path** and a throw there would either 5xx the user or (if caught) fail to record the increment — undercounting exactly at the boundary the cap most needs to hold.

**Required fix (MUST, for the plan):** clamp the TTL to the KV minimum:
```ts
const ttl = Math.max(60, secondsToNextLocalMidnight + BUFFER);
```
The "+ small buffer" the spec already mentions does not solve this unless the buffer itself is ≥60s at all times — a 60s floor via `Math.max` is the clean guarantee. Effect: a key written in the last minute lives ~60s into the next day (overlapping the new day's key briefly) — harmless, since the key includes `{localDate}` so the new day uses a *different* key anyway. The stale old-day key just lingers a few extra seconds.

### 2b — KV eventual consistency / global propagation — CONFIRMED (no worse surprise than spec already accepts, with one nuance to document)

Confirmed from CF "How KV works": writes are **usually** immediately visible at the **writing** PoP, but this is **not guaranteed**; cross-PoP propagation can take **up to ~60s** (cached negative lookups included). The spec already accepts non-atomic read-modify-write and ~2× burst across the boundary.

- **No catastrophic undercount:** the runtime requirement that matters — a device's repeated searches within a short window usually hit the **same** PoP — means read-after-write is *usually* fresh for the dominant single-user-hammering case the cap targets. The eventual-consistency window does **not** cause "badly undercount" in normal usage.
- **Worse-than-2× nuance to write down (not new, but sharpen it):** if a single device's requests land on **different PoPs** (mobile network handoff, anycast reroute) within the ~60s propagation window, each PoP can independently read a stale lower count → the realistic worst case is **not strictly 2×** but "~2× *per concurrently-stale PoP*." For a consumer mobile app a single device rarely spans multiple PoPs simultaneously, so practically it stays near the spec's ~2× estimate. Acceptable for cost-shaping; the DO seam remains the correct upgrade for any future hard/paid enforcement. Recommend the spec's §7 "Concurrency" note explicitly say "per-PoP staleness, not a global 2× ceiling" so nobody later mistakes it for a hard bound.
- **1-write-per-second-per-key limit** also exists (CF returns 429 on >1 write/s to the *same* key). A single human triggering searches won't approach 1/s; not a concern, but worth a one-line awareness note since the key is per-device-per-day.

### 2c — `ctx.waitUntil` for best-effort metrics — CONFIRMED

`ctx.waitUntil(promise)` is the correct primitive for fire-and-forget metric writes that must not block or fail the response. It **must** be threaded from the handler signature `fetch(request, env, ctx)` (or, with the module-worker `ExecutionContext`, passed down to wherever the metric write happens). Errors inside the awaited promise do not surface to the client — matching the spec's "errors swallowed, a metric write must never 5xx a user request." Confirmed correct. Implementation note: thread `ctx` explicitly into `meterSearch`/telemetry helpers; do not reach for a global.

### 3 — `Date.now()` in workerd — CONFIRMED

`Date.now()` and `new Date()` are available in request scope. workerd freezes the clock between I/O for the well-known timing-side-channel mitigation (the wall clock advances only on I/O), which is **irrelevant** to a date-bucket counter — it does not affect `Intl` formatting or the date computation. `Math.floor(Date.now()/1000)` as the injectable default is correct and idiomatic.

### 4 — RN/Expo 429 header access — CONFIRMED (one casing nuance)

RN's `fetch` `Response.headers` is a WHATWG `Headers` object: `res.headers.get('Retry-After')` / `res.headers.get('x-ratelimit-remaining')` work, and **`Headers.get` is case-insensitive** per spec — both RN's implementation and Hermes honor this. No gotcha for reading the custom `X-RateLimit-*` / `Retry-After`. Two minor notes (non-blocking):
- The spec's primary data path is the **429 JSON body** (`limit`/`used`/`reset_at_unix`), not headers — that's the more robust choice; body parsing has no casing/CORS concern. Good.
- If headers were ever the source of truth in a *web* context (they aren't here — native), `X-RateLimit-*` custom headers would need `Access-Control-Expose-Headers`. Native RN `fetch` is **not** subject to browser CORS, so exposing custom headers is unrestricted. No action.

### 5 — vitest-pool-workers: KV + Intl tz + `ctx.waitUntil` + deterministic time — CONFIRMED, with a constraint that *validates the spec's design choice*

Confirmed from CF Vitest-integration docs:
- `@cloudflare/vitest-pool-workers` runs tests **inside workerd**, so `Intl` behaves exactly as production (full ICU, real tz math) — the tz/DST tests run against the real runtime, not a Node shim. This is the right gate.
- KV is testable via a configured test namespace (`env.NAMESPACE`); `import { env } from 'cloudflare:test'`.
- `ctx.waitUntil` is testable: create context with `createExecutionContext()`, run the handler, then **`await waitOnExecutionContext(ctx)`** before asserting on the side-effect KV writes (metrics). This is exactly how to test §8 telemetry deterministically.
- **Key constraint that confirms the spec's `now`-injection decision:** *Vitest fake timers do NOT apply to the KV/R2/cache simulators* — you cannot expire a KV key by advancing fake time, and faking `Date.now()` is brittle across the worker boundary. **Therefore the spec's choice to make `now` an injectable parameter (default `Math.floor(Date.now()/1000)`) is not just convenient — it is the *correct and necessary* way to get deterministic date/DST/TTL tests here.** Passing explicit `now` + explicit `tz` sidesteps both the fake-timer limitation and the `TZ=UTC` ambient-zone issue. The spec got this right; do **not** try to test via `vi.useFakeTimers()` for the KV-expiry or local-date logic.
  - Corollary: you cannot assert real TTL *expiry* by advancing time. Test the TTL **value** by asserting the `expirationTtl` passed to `put` (spy/mocked binding or read-back of options) and unit-test `nextLocalMidnightUnix`/`ttl` arithmetic directly with injected `now` — don't try to observe the key actually vanishing.

## 5. High-Priority Findings 🟠

**🟠 H1 — KV `expirationTtl` 60s floor will throw on near-midnight writes (Finding 2a).**
The spec's TTL formula ("seconds until next local midnight + small buffer") produces sub-60s values in the final ~60s before local midnight, and KV `put` rejects any `expirationTtl < 60`. On the *counter write* (request path) this is a real failure, not a swallowed metric. **Fix is one line:** `Math.max(60, secondsToNextMidnight + buffer)`. Must be in the plan and must have a test (write a counter at 23:59:30-local, assert `put` is called with `expirationTtl >= 60` and does not throw). Cheap; high consequence if missed.

## 6. Medium Findings 🟡

**🟡 M1 — DST offset must be read at the candidate instant, not "now" (Finding 1d).** Intl-only is sufficient; do NOT subtract today's offset from a naive tomorrow-midnight. Use the offset-at-candidate + one settle iteration (snippet above). The spec already mandates a DST-boundary test — keep it; the snippet is the implementation it should pass.

**🟡 M2 — Eventual-consistency worst case is "~2× per stale PoP," not a hard 2× (Finding 2b).** Sharpen §7's concurrency note so the soft-cap bound isn't later mistaken for a guarantee. No code change; documentation precision.

**🟡 M3 — Test TTL by asserting the `put` option value, not by observing expiry (Finding 5).** vitest-pool-workers fake timers don't expire KV keys. Plan's test for "TTL set" must assert the argument, not the eviction.

## 7. Design Constraints for the Plan (non-negotiable)

- **MUST** clamp counter TTL: `expirationTtl = Math.max(60, secondsToNextLocalMidnight + buffer)` — KV rejects <60s. (H1)
- **MUST** compute next-local-midnight using the offset **at the candidate instant** with a single settle iteration; Intl-only, no date lib needed. (M1)
- **MUST** pass an explicit `timeZone` to every `Intl` call; never rely on workerd's ambient `TZ=UTC`. (1a)
- **MUST** keep `now` as an injectable param (default `Math.floor(Date.now()/1000)`) — it is the *only* deterministic path for date/DST/TTL tests; fake timers do not work for KV/Intl-tz. (5)
- **MUST** test telemetry side-effects via `createExecutionContext()` + `await waitOnExecutionContext(ctx)` before asserting KV metric writes. (2c, 5)
- **MUST** test TTL by asserting the `expirationTtl` value passed to `put`, not by advancing time to expire the key. (M3)
- **MUST NOT** introduce a date/tz dependency (luxon/date-fns-tz/Temporal-polyfill) in the Worker — Intl + the snippet above is sufficient and the spec's "no date lib in Worker" stance is validated. A dependency is NOT warranted.
- **SHOULD** prefer `timeZoneName:'longOffset'` over `'shortOffset'` if parsing an offset string anywhere (padded `±HH:MM`, no `GMT` special-case). (1c)
- **SHOULD** document the eventual-consistency bound as "per-PoP staleness," not a global 2× ceiling. (M2)
- **SHOULD** read 429 data from the **JSON body** (already the plan), not headers — avoids any casing/exposure concern entirely. Header reads (`Headers.get`) are case-insensitive in RN anyway. (4)
